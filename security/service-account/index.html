    ServiceAccount

    Los ServiceAccount sirven para dar una identidad a los procesos que se ejecuten dentro de un Pod, y de esta manera poder darles permisos sobre el cluster.

    Por defecto, en todos los Namespace existe un ServiceAccount de nombre default con los permisos mínimos que puede necesitar un Pod para interactuar con la API de Kubernetes.

    Si en la definición del Pod no se especifica la propiedad spec.serviceAccountName, se utilizará el default del Namespace.

    Con la creación de cada ServiceAccount, Kubernetes crea un Secret asociado, que contendrá un token que se habrá generado, para que el ServiceAccount pueda autenticarse contra la API de Kubernetes. Este Secret se monta en cada uno de los Pods que utilicen el ServiceAccount.

    Notas

    • Se puede desactivar el uso por defecto del ServiceAccount default en los Pods que no tengan definida la propiedad spec.serviceAccountName, añadiendo la propiedad automountServiceAccountToken: false.
    • Si en un Pod se añade la propiedad automountServiceAccountToken: false, el Secret que contiene el token del ServiceAccount no se montará automáticamente.

    Ejemplo

    apiVersion: v1
kind: ServiceAccount
metadata:
  name: build-robot

    Añadir ImagePullSecrets al ServiceAccount

    Se pueden añadir ImagePullSecrets a un ServiceAccount para que todo Pod que utilice el ServiceAccount tenga acceso a esos Secret para poder realizar los pull de las imagenes Docker.

    Para ello hay que configurar la stanza imagePullSecrets.

    Ejemplo

    apiVersion: v1
kind: ServiceAccount
metadata:
  name: build-robot
imagePullSecrets:
- name: myregistrykey

    Declaración imperativa con kubectl

    kubectl create serviceaccount my-service-account
