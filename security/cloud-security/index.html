    Cloud Native Security

    Las 4 C's de la seguridad Cloud Native

    Este modelo divide la seguridad en cuatro capas diferentes. Cada una de ellas se sustenta en las capas más exteriores del modelo, de tal forma que no tiene sentido enforcar grandes esfuerzos en securizar una de las capas más internas, si las externas están mas expuestas a ataques.

    Cloud

    Esta capa representa toda la infraestructura en la que se sustenta la computación de Kubernetes, ya sea en un Data Center propio o en un proveedor Cloud. Si esta capa no es segura, el resto van a ser vulnerables a ataques.

    Recomendaciones

    Concepto a revisar Recomendación
    Acceso red al API Server El acceso al Control Plane de Kubernetes no debería ser público. Habría que permitir el acceso solo desde un listado de IP's concreto
    Acceso red a los nodos A ser posible los nodos no deberían tener acceso público. Deberían permitir únicamente conexiones a los puertos que requiera el Control Plane y las conexiones de los Service de tipo NodePort y LoadBalancer.
    Acceso de Kubernetes a la API del Cloud Provider Limitar el acceso que tiene Kubernetes únicamente a los recursos del proveedor Cloud que requiere gestionar.
    Acceso al etcd El acceso al etcd debe estar limitado únicamente al Control Plane de Kubernetes. Dependiendo de la configuración, el etcd se debería ser accedido via HTTPS.
    Encriptación del etcd Dado que el etcd almacena el estado de los API objects (incluyendo los Secrets), se deberían de encriptar los datos (encryption at rest).

    Cluster

    A nivel de cluster hay dos areas de actuación:

    • Securizar los componentes del cluster que son configurables.
    • Securizar las aplicaciones que se ejecutan en el cluster

    Securizar componentes del cluster

    Recomendaciones

    API de Kubernetes

    Recomendaciones
    Realizar todas las conexiones a la API de Kubernetes a través de HTTPS
    Habilitar un mecanismo de autenticación para hacer uso de la API
    Habilitar un mecanismo de autorización para configurar qué endpoints de la API se pueden utilizar

    Kubelet

    Recomendaciones
    Por defecto el kubelet permite realizar llamadas a la API sin necesidad de autenticarse. Habría que habilitar un mecanismo de autenticación y autorización para poder hacer uso de la API del kubelet

    Capabilities en tiempo de ejecución

    Recomendaciones
    Limitar el uso de recursos del cluster mediante Resource Quota y Limit Ranges
    Controlar mediante Security Context o Security Policies, los privilegios con los que se ejecutan los contenedores
    Evitar que el kernel cargue módulos innecesarios que puedan suponer una vulnerabilidad. Esto se puede evitar por ejemplo con SELinux
    Restringir el acceso a red mediante Network Policies
    Limitar el acceso a la API de de metadatos del Cloud, para que solo ciertos Pods tengan acceso a los datos sensibles
    Controlar a qué nodos tienen acceso los Pods (en qué nodos pueden desplegarse), mediante el uso de taints o nodeSelectors.

    Securizar componentes de las aplicaciones

    Recomendaciones

    Concepto a revisar Recomendación
    Acceso a la API de Kubernetes Limitar mediante la autorización RBAC, qué operaciones pueden realizar ciertos usuarios o aplicaciones, sobre los recursos de Kubernetes (API objects).
    Autenticación Gestionar correctamente la autenticación de los usuarios (dado que no se almacenan datos del usuario, no existe un API object que lo represente) y de las aplicaciones (Service Accounts).
    Gestionar datos sensibles de las aplicaciones Almacenar datos sensibles de las aplicaciones en Secrets y encriptarlos en el etcd (encryption at rest).
    Admisión de Pods Crear Pod Security Policies que definan qué reglas deben cumplir los Pods para poder ser admitidos en el sistema.
    Recursos de los Pods Limitar el uso de recursos que pueden hacer los Pods mediante request y limits, y establecer diferentes clases de QoS en función de la criticidad del Pod.
    Acceso red de los Pods Limitar mediante Network Policies las conexiones que pueden admitir y/o realizar los Pods.
    Securizar Ingress Exponer el Ingress via HTTPS para securizar las conexiones que se realicen desde el exterior del cluster.

    Container

    Recomendaciones

    Concepto a revisar Recomendación
    Vulnerabilidades del contenedor y dependencias del Sistema Operativo En el proceso de construcción de la imágen, escanear el contenedor en búsqueda de vulnerabilidades.
    Firmar imágenes Firmar las imágenes de contenedor, para confiar únicamente en los proveedores de imágenes de confianza.
    Usuarios con privilegios Limitar los permisos que tienen los usuarios de los contenedores, para que sólo puedan ejecutar las operaciones necesarias para el correcto funcionamiento de las aplicaciones.

    Code

    Recomendaciones

    Concepto a revisar Recomendación
    Securizar conexiones Las aplicaciones sólo deberían establecer conexiones a través de TLS. Adicionalmente, se puede encriptar el tráfico de red entre dos servicios con mTLS.
    Exposición de puertos Limitar el rango de puertos expuestos, únicamente a aquellos que sean esencialmente necesarios para que opere la aplicación o para que sea monitorizada.
    Vulnerabilidades en librerías de terceros Escanear regularmente las librerías de terceros en búsqueda de vulnerabilidades.
    Análisis de código estático Mediante una herramienta de análisis de código estático, analizar el código fuente en búsqueda de malas practicas.
    Ataques dinámicos Utilizar herramientas para automatizar pruebas de ataques dinámicos contra las aplicaciones, tales como SQL Injection, CSR o XSS.
