    Debugging

    Troubleshooting de aplicaciones

    Debugging Pods

    El primer paso para debugar una aplicación, es ver el estado del Pod y los eventos acontecidos recientemente.

    Mediante el comando kubectl describe pod <pod-name>, se puede obtener información general del Pod:

    • Configuración de los contenedores
    • Estado actual del Pod
    • Eventos del Pod

    Nota

    Una forma de obtener información más detallada acerca de un Pod es mediante el comando kubectl get pod <pod-name> -o yaml. De esta forma se puede obtener en formato YAML toda la información que hay almacenada en Kubernetes sobre ese Pod.

    Información

    Posibles estados del Pod y motivos:

    • Pending: significa que el Pod no puede ser emplazado en un nodo. Esto puede suceder porque no hay recursos o nodos suficientes, o de utilizar la propiedad hostPort, que no hay nodos con ese puerto disponible. Solución: Eliminar algunos Pods, ajustar las requests de recursos de los Pods o añadir nuevos nodos.

    • Waiting: el Pod se ha desplegado en un nodo, pero no puede ser ejecutado. Normalmente la causa más común de esto es que el Pod no haya podido hacer pull de la imagen Docker, correctamente. Solución: Habría que revisar que el nombre de la imagen sea la correcta, que la imagen exista en el repositorio o probar manualmente de realizar un docker pull <image> en el nodo para comprobar que la imagen pueda ser obtenida.

    Pod no arranca correctamente
    Si el Pod falla o no está en estado healthy, hay que analizar los logs de las aplicaciones para buscar más información. Para ello ejecutar:

    kubectl logs <pod-name> <container-name>

## Para ver los logs de un contenedor que haya fallado anteriormente
kubectl logs --previous <pod-name> <container-name>
    Otra forma para debugar el Pod es accediendo a sus contenedores mediante una Shell, si estos disponen de debugging utilities que puedan ser utilizadas (p.ej. realizar un cat sobre un fichero concreto):

    kubectl exec -it cassandra -- sh
    Pod no tiene comportamiento esperado

    Si el problema es que el Pod no se ejecuta como se esperaba, es posible que haya algún tipo de error en su descriptor de despliegue que haga que no se ejecute correctamente (p.ej. si hay algún error tipográfico, esa propiedad es ignorada).

    Para validar si se trata de un error con el descriptor, habría que eliminar el Pod y volver a ejecutarlo con la opción --validate:

    kubectl apply --validate -f mypod.yaml

    Si hubiera algún tipo de problema, saltaría un mensaje de error informando de qué está mal.

    Otra comprobación sería obtener la descripción del Pod desplegado y compararla con la del descriptor para ver si existe alguna diferencia (sin tener en cuenta los campos que añade el servidor de la API. p.ej. startTime).

    kubectl get pods/mypod -o yaml > mypod-on-apiserver.yaml

    Contenedores efímeros (alpha)

    Cuando acceder via Shell a dentro de un contenedor no sea posible o suficiente (p.ej. porque la imagen que se esté utilizando no disponga de debugging utilities), otra alternativa son los contenedores efímeros.

    Estos contenedores se levantarán dentro del mismo Pod y compartirán el mismo "namespace de procesos". De esta forma, el nuevo contenedor compartirá el mismo arbol de procesos y sistema de ficheros que el contenedor de la aplicación.

    El comando a ejecutar para levantar un contenedor efímero:

    kubectl alpha debug -it ephemeral-demo --image=busybox --target=<app-container-name>

    Acceder logs de Init Container

    Para leer los logs de un Init Container ejecutar:

    kubectl logs <pod-name> -c <init-container-name>

    Si los contenedores ejecutan un script bash, se puede ejecutar set -x al inicio del script para que se graben en el logs los comandos que se van a ir ejecutando.

    Mensaje de terminación

    Cuando un contenedor termina su ejecución (ya sea porque ha terminado de ejecutar su proceso o por un error), vuelca un "mensaje de terminación" sobre el fichero /dev/termination-log. Si el contenedor volcara el mensaje en otro fichero, se puede indicar en la definición del Pod con la propiedad spec.containers[].terminationMessagePath.

    Adicionalmente se puede añadir la propiedad spec.containers[].terminationMessagePolicy: "FallbackToLogsOnError", para que si el contenedor termina con error y el fichero del "mensaje de terminación" está vacío, utilice el último bloque de logs del contenedor como "mensaje de terminación".

    Debugging Replication Controllers

    Dado que los Replication Controllers solo se encargan de gestionar el número de Pods desplegados, para debugarlos únicamente hay que debugar los Pods.

    Adicionalmente se pueden observar los eventos de los Replica Set para ver si hubiera algún problema a la hora de crear nuevas instancias:

    kubectl describe rs <replica-set-name>

    Debugging Services

    Para empezar a debugar un Service hay que revisar que tenga el número de Endpoints asociados correcto. Debe coincidir con el número de Pods del Service. Para comprobarlo:

    kubectl get endpoints <service-name>

    Si el comando anterior no devolviera el mismo número de direcciones IP que de Pods, habría que revisar que se obtenga el listado correcto de Pods al buscar por las labels del Service:

    kubectl get pods --selector=label1=value1,label2=value2
    Si el listado de Pods es el correcto, es posible que no se estén exponiendo los puertos correctos. Habría que comprobar que los containerPort de los Pod coinciden con los targetPort del Service.

    Si por el contrario, los Endpoints si tienen IP's, pero una vez se ha establecido conexión con el Service, la conexión se cae, es probable que el Pod no esté funcionando correctamente. En ese caso habría que revisar el contador de reinicios del Pod.

    Otra comprobación sería intentar acceder al Pod directamente a través de su dirección IP.

    Si en ningún caso la aplicación es accesible, es posible que la aplicación no esté exponiendo el mismo puerto que haya configurado en el Pod. Para comprobarlo ejecutar desde dentro el cluster:

    wget -gO- <pod-IP>:<port>

    El Service no resuelve por DNS

    Para comprobar que el dominio del Service resuelve correctamente, habría que utilizar la funcionalidad nslookup desde un Pod en el mismo Namespace que la aplicación.

    nslookup <service-name>

    Si fallara, podria ser que Pod y Service estubieran en Namespaces diferentes. Para ello comprobar:

    nslookup <service-name>.<service-namespace>

    Si aún así fallara, habría quer comprobar el FQDN del Service:

    nslookup <service-name>.<service-namespace>.svc.cluster.local
    Si resuelve correctamente, habría que comprobar el fichero /etc/resolv.conf del Pod para ver que sea correcto:

    • Debe contener una entrada nameserver con la IP del servicio DNS del cluster.
    • Debe contener una entrada search con todos los sufijos del cluster:
      • <service-name>.<service-namespace>.svc.cluster.local para encontrar los Service del mismo Namespace.
      • svc.cluster.local para encontrar los Service de cualquier Namespace.
      • cluster.local para encontrar dominios del cluster.
    • Debe contener una entrada options con la propiedad ndots con un valor suficientemente alto como para que el servidor DNS considere los search paths.

    Si aún asi el comando fallara, sólo quedaría comprobar el dominio de la API de Kubernetes, ya que este siempre debería funcionar:

    nslookup kubernetes.default

    Si este comando falla, habria que comprobar que el kube-proxy funcione correctamente.

    El Service no funciona por IP

    Comprobar que el Service es accesible via IP. La dirección IP se puede obtener con el comando kubectl get svc <service-name>. Para comprobar que se puede acceder via IP ejecutar desde dentro del cluster:

    wget -qO- <service-IP>:<port>

    Si no funcionara, habría que revisar que el Service esté correctamente configurado con los puertos y protocolos correctos y que los Pods se estén ejecutando correctamente.

    Comprobar que kube-proxy funciona correctamente

    Si los Endpoints y la configuración de los Services es correcta, es probable que el problema esté en la implementación que se ejecute por debajo de los Services. En la mayoria de clusters Kubernetes será kube-proxy.

    Primero habría que comprobar que el proceso de kube-proxy se esté ejecutando en el nodo. Para ello, habrá que acceder via SSH y ejecutar el siguiente comando:

    ps auxw | grep kube-proxy

    Si el proceso se está ejecutando, habria que comprobar en los logs, si está teniendo dificultades para contactar al _master. Se puede revisar el fichero /var/log/kube-proxy.log.

    Si aparecen mensajes de error habría que revisar la configuración del nodo y revisar si el proceso de instalación se ha realizado correctamente. Si la instalación se ha realizado manualmente, debería de haberse instalado el binario conntrack, por lo que habría que revisar que se haya instalado correctamente.

    En función del modo de ejecución del kube-proxy (iptables o ipvs), habría que revisar la tabla de redirecciones correspondiente para comprobar que las entradas son correctas:

    ## iptables
iptables-save | grep <service-name> # -> Debe devolver una regla KUBE-SERVICES y una cadena KUBE-SVC-<hash>

## ipvs
ipvsadm -ln # -> Debe devolver una entrada con la <service-IP>:<service-port> del Service y tantas entradas <pod-IP>:<pod-port> como Pods tenga el Service

    Cuando el kube-proxy opera en modo iptables, se puede dar el caso que un Pod no pueda accederse a si mismo a través de la IP del Service. En ese caso habría que revisar que el flag hairpin-mode de kube-proxy esté correctamente configurado (ver Documentación para más detalle).

    Debugging del cluster

    Se puede obtener el estado actual de los nodos con el comando kubectl get nodes.

    Mediante el comando kubectl describe node <node-name>, se puede obtener información general del nodo:

    • Configuración del nodo
    • Estado actual del nodo
    • Capacidad del nodo
    • Recursos disponibles y en uso
    • Eventos del nodo

    Nota

    Una forma de obtener información más detallada acerca de un nodo es mediante el comando kubectl get node <node-name> -o yaml. De esta forma se puede obtener en formato YAML toda la información que hay almacenada en Kubernetes sobre ese nodo.

    Si se desea obtener información más detallada del estado global del cluster se puede ejecutar el comando:

    kubectl cluster-info dump

    Advertencia

    Este comando devuelve TODA a información almacenada en el cluster (descriptores, logs, eventos...), por lo que hayq que realizar un proceso de búsqueda para encontrar la información que se desea.

    Para comprobar que todos los componentes del cluster operan correctamente hay que ir nodo por nodo observando los diferentes logs:

    Nodos master - API server: /var/log/kube-apiserver.log - Scheduler: /var/log/kube-scheduler.log - Controller: /var/log/kube-controller-manager.log

    Nodos worker - kubelet: /var/log/kubelet.log - kube-proxy: /var/log/kube-proxy.log
