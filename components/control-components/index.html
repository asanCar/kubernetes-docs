    Componentes del Control Plane

    Componentes del Control Plane

    Estos componentes son los encargados de tomar decisiones globales sobre el cluster (p.ej. scheduling), asi como detectar y responder ante eventos (p.ej. si una replica de un Pod se cae, levantar una nueva replica).

    Componentes:

    • kube-apiserver
    • etcd
    • kube-scheduler
    • kube-controller-manager
    • cloud-controller-manager

    kube-apiserver

    Es la pieza principal del cluster Kubernetes. Es el componente encargado de exponer la API de Kubernetes y actúa como front end del Control Plane. Se trata del único componente que interactúa directamente con etcd.

    Esta pensado para que escale horizontalmente, por lo que pueden existir múltiples instancias a la vez y balancear las peticiones que reciba el cluster.

    Ejemplo del proceso de creación de un nuevo Pod

    1. Autenticación
    2. Autorización
    3. Obtener el estado actual del etcd
    4. Se crea el objeto Pod pero sin alojarse en ningún worker
    5. Se actualiza el estado del etcd y se notifica al usuario que el Pod se ha creado
    6. El Scheduler que constantemente esta observando al kube-apiserver, observa que hay un nuevo Pod sin asignar. Busca e identifica un nodo en el que poder desplegar el Pod y se lo notifica al kube-apiserver.
    7. El kube-apiserver actualiza el etcd y ordena al kubelet del woker seleccionado que cree un nuevo Pod.
    8. El kubelet ordena al Container Runtime Engine que cree un nuevo contenedor con la imagen especificada.
    9. Una vez creado el contenedor, el kubelet notifica al kube-apiserver, el cual a su vez actualiza el etcd con la nueva información.

    etcd

    Base de datos clave-valor en alta disponibilidad utilizada para almacenar todos los datos del cluster. Toda la información que devuelve el comando kubectl get [node|pod|secret...] procede del etcd. Cualquier cambio que se aplique sobre el cluster, se considerará finalizado cuando se acabe de actualizarse etcd.

    etcd se puede instalar directamente sobre una máquina del clúster, para ejecutarse como servicio, o desplegarse como contenedor (p.ej. con kubeadm).

    kube-scheduler

    Componente encargado de observar los Pods recién creados aún no tengan un nodo asignado y seleccionar un nodo disponible que cumpla los requisitos, para que sea desplegado en él.

    Cuidado

    Solo se encarga de decidir en qué nodo deben ir los Pods. Quien se encarga de crearlos son los kubelet.

    Para tomar la decisión se basa en factores como el uso de recursos que requiere el Pod, los recursos disponibles en el nodo, especificaciones de affinity y anti-affinity, localidad de los datos, restricciones por software, hardware, políticas...

    Proceso de elección de nodo

    1. Filtrado de nodos: en función de los requisitos de memoria y CPU de un Pod (u otros requisitos), se filtrará el listado de nodos disponibles del cluster para quedarse únicamente con los que cumplan los requisitos del Pod.
    2. Ordenar por prioridad: en función de los recursos disponibles de los nodos, se ordena el listado, puntuando los nodos de 0 a 10, dando mayor puntuación a los que una vez alojaran el Pod, les quedara más recursos disponibles.

    kube-controller-manager

    Este componente representado en un único binario, se encarga de ejecutar varios procesos de control, encargados de observar constantemente el estado del cluster mediante el kube-apiserver y aplicar cambios en el estado actual para conseguir el estado deseado.

    Los Controllers que incluye son:

    • Node Controller: Responsable de notificar y actuar cuando algún nodo se cae. Por defecto, monitoriza los nodos cada 5 segundos.

      Si algún nodo no responde, se le da un periodo de gracia de 40 segundos. Si pasado este tiempo no responde, se marca el nodo como UNREACHABLE. Si al cabo de 5 minutos el nodo no se ha recuperado, los Pods que formaran parte de un ReplicaSet, se realojarán en el resto de nodos disponibles.

    • Replication Controller: Responsable de mantener el número correcto de Pods indicado en cada uno de los objetos "Replication Controller" del sistema.
    • Endpoint Controller: Genera los objetos Endpoints (para unir Services con Pods).
    • Service Account & Token Controllers: Responsable de generar las cuentas y API access tokens por defecto de los nuevos Namespaces.

    cloud-controller-manager

    Este componente únicamente se utiliza cuando se despliega Kubernetes en un proveedor Cloud. Contiene lógica de control específica del proveedor Cloud para integrarse con su API y gestionar los componentes que interactúan con el proveedor Cloud de forma independiente, respecto al resto de componentes que únicamente interactúan con el cluster Kubernetes.
