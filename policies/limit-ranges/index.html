    Limit Ranges

    Por defecto, los contenedores se ejecutan dentro del cluster de Kubernetes sin ningún tipo de restricción en el consumo de recursos.

    Mediante el uso del API object LimitRange, los administradores pueden aplicar las siguientes restricciones:

    • Regular el uso mínimo y máximo de recursos por Pod o Container en un Namespace.
    • Regular el almacenamiento mínimo y máximo que pueden solicitar los PersistentVolumeClaim de un Namespace.
    • Configurar request/limit por defecto para los contenedores de un Namespace.

    Notas

    • El Admission Controller LimitRanger se encarga de inyectar en tiempo de creación, request/limit por defecto, en los contenedores que no los lleven configurados.
    • Cualquier petición para crear/actualizar un recurso, que vulnere las restricciones definidas en un LimitRange, será denegada con un "403 Forbidden".
    • Las validaciones del LimitRange se realizan en la fase de "Pod Admission", por lo que no se aplica sobre los Pods que ya se estén ejecutando. Sucede lo mismo cuando se modifica un LimitRange: no se aplican cambios en los Pods ya creados.

    Recursos por defecto

    Se pueden utilizar los LimitRange para modificar en tiempo de creación, la definición de los contenedores, para inyectar requests/limits por defecto.

    Asignar valores por defecto para los limit de memoria, permite que un Pod pueda ser desplegado en un Namespace con un ResourceQuota definido.

    Ejemplo

    LimitRange que configura un request de 0.5 CPU y un limit de 1 CPU para los contenedores del Namespace "my-test" que no los tengan configurados: 

    apiVersion: v1
kind: LimitRange
metadata:
  name: cpu-limit-range
  namespace: my-test
spec:
  limits:
  - default:
      cpu: 1
    defaultRequest:
      cpu: 0.5
    type: Container

    Si se intenta crear un Pod en un Namespace con este LimitRange de ejemplo, se pueden dar estos casos (similar para memoria RAM):

    • Pod sin requests/limits: Se creará un Pod cuyos contenedores tendrán un request de 0.5 CPU y un limit de 1 CPU.
    • Pod con limit de 2 CPU y sin request: Se creará un Pod cuyos contenedores tendrán un request de 2 CPU y un limit de 2 CPU.
    • Pod con request de 0.2 CPU y sin limit: Se creará un Pod cuyos contenedores tendrán un request de 0.2 CPU y un limit de 1 CPU.

    CPU/Memoria Min y Max

    Se puede limitar qué Pods pueden ser admitidos en un Namespace, en función de los requests/limits que tengan definidos.

    Por ejemplo no permitir desplegar Pods que no lleguen a un mínimo de recursos o que no superen un máximo.

    Ejemplo

    LimitRange que solo admite contenedores que consuman mínimo 500Mi de RAM y no superen el uso de 1Gi de RAM: 

    apiVersion: v1
kind: LimitRange
metadata:
  name: mem-min-max-demo-lr
spec:
  limits:
  - max:
      memory: 1Gi
    min:
      memory: 500Mi
    type: Container

    El LimitRange del ejemplo, al no especificar "valores por defecto" para los contenedores que no definan requests/limits, toma por defecto el valor máximo del limit. Es decir, seria lo mismo que crear el siguiente LimitRange:

    apiVersion: v1
    kind: LimitRange
    metadata:
      name: mem-min-max-demo-lr
      namespace: my-test
    spec:
      limits:
      - default:
          memory: 1Gi
        defaultRequest:
          memory: 1Gi
        max:
          memory: 1Gi
        min:
          memory: 500Mi
        type: Container

    Nota

    Especificar valores máximos de recursos es útil para evitar desplegar Pods que requieran más recursos de los que proporcionan los nodos.

    Si se intenta crear un Pod en un Namespace con este LimitRange de ejemplo, se pueden dar estos casos (similar para CPU):

    • Pod sin requests/limits: Se creará un Pod cuyos contenedores tendrán un request de 1Gi RAM y un limit de 1 Gi RAM.
    • Pod con request de 500Mi y limit de 2Gi: No se permitirá el despliegue dado que sobrepasa el máximo permitido en el Namespace.
    • Pod con request de 200Mi y limit de 1Gi: No se permitirá el despliegue dado que no llega al mínimo requerido en el Namespace.

    Almacenamiento Min y Max

    Al igual que con los recursos CPU y memoria RAM para los contenedores, se puede limitar qué PersistentVolumeClaim se admiten en un Namespace, en función del almacenamiento que se solicite.

    Esto permite a los administradores controlar por ejemplo el máximo de almacenamiento que puede solicitar un único Claim.

    Ejemplo

    apiVersion: v1
kind: LimitRange
metadata:
  name: storagelimits
spec:
  limits:
  - type: PersistentVolumeClaim
    max:
      storage: 2Gi
    min:
      storage: 1Gi
